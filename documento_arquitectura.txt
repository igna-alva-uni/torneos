
DOCUMENTO DE ARQUITECTURA Y CONTRATO DE MICROSERVICIOS1. Descripción generalEste proyecto consiste en el desarrollo de una plataforma de gestión de torneos de videojuegos basada en arquitectura de microservicios.Cada microservicio es responsable de un dominio específico y se comunica con otros mediante llamadas REST.2. Arquitectura generalArquitectura: Microservicios Comunicación: REST Descubrimiento de servicios: Eureka Base de datos: 1 por microservicio Lenguaje: (Java + Spring Boot) 3. Estructura de URLsTodas las APIs siguen el formato:/api/v1/{servicio}/{recurso}Ejemplos:/api/v1/usuarios/usuarios/api/v1/equipos/equipos/api/v1/torneos/torneos4. Endpoints REST estándarCada recurso debe implementar como mínimo:Obtener todos: GET    /{recurso}Buscar por id: GET    /{recurso}/{id}crear: POST   /{recurso}actualizar: PUT   /{recurso}/{id}eliminar por id: DELETE /{recurso}/{id}5. Formato de respuestasLas respuestas serán en formato JSON sin wrapper.Ejemplo (objeto):{  "id_usuario": 1,  "username": "properolol",  "email": "test@gmail.com"}Ejemplo (lista):[  {    "id_usuario": 1,    "username": "properolol",    "email": "test@gmail"  },  {    "id_usuario": 2,    "username": "lleyo",    "email": "testeo@gmail"  }]6. Manejo de errores (por ver)Formato estándar:{  "codigo": "USUARIO_NO_ENCONTRADO",  "mensaje": "El usuario no existe"}Códigos HTTP:400 → datos inválidos 404 → recurso no encontrado 500 → error interno 7. Convenciones de nombresIdioma: Español Formato: camelCase: para javasnake_case: para sql Ejemplos:idUsuario                                                                                                                                         id_usuarionombreEquipo                                                                                                                         nombre_equipofechaCreacion                                                                                                                                        fecha_creacion8. Comunicación entre microserviciosRegla principal:"from service X" = validación mediante llamada RESTEjemplo:Si un campo referencia a otro servicio:id_usuario → from service USER SERVICESe debe validar con:GET /api/v1/usuarios/usuarios/{id}Flujo de validación (por ver):1. Se recibe request con id_usuario2. Se llama a USER SERVICE3. Si responde 200 → continuar4. Si responde 404 → error5. Si no responde → retry🔁 Política de reintento (por ver)Se realiza la llamada REST Se espera máximo 5 segundos Si falla → reintento automático (1 vez) Si vuelve a fallar → error personalizado 9. Organización de recursosCada tabla corresponde a un endpoint independiente No se mezclan entidades en un mismo endpoint: solo se devuelve la información de la tabla requerida (Esto es una simplificación. En un sistema real, no todas las tablas son expuestas directamente.)10.
